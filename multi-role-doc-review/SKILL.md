# 多角色文档审查

触发条件：用户要求多角色/多视角审查文档、交叉验证方案、或提到"角色审查"。

## 完整流程

### 步骤 1：读取文档

读取用户指定的文档全文。若文档超过 5000 字，询问用户是全文审查还是指定章节。

### 步骤 2：角色选择

使用 clarify 工具的多选模式，呈现默认角色供用户选择：

```
请选择参与审查的角色（可多选，也可自行添加）：

1. 产品经理
2. UI/UX 设计师
3. 架构师
4. 开发工程师
```

用户可以取消某些角色或追加自定义角色。确认后进入下一步。

### 步骤 3：并行独立评审

为每个选定的角色调用 delegate_task 并行执行。每个子任务的 prompt 结构：

```
你是一个 [角色名]，正在审查一份项目文档。

## 你的身份
[角色身份描述]

## 你的关注维度
[关注维度清单]

## 你的评判红线
[红线规则]

## 文档内容
[完整文档或指定章节]

## 输出要求

请严格按照以下格式输出：

### 整体评价
一句话定性：可行 / 需要大改 / 方向错误

### 具体问题
按严重程度排序（高→低），每条必须：
- 引用文档原文（用 > 引用格式）
- 说明问题本质
- 标注严重程度：🔴 高 / 🟡 中 / 🟢 低

### 优化建议
与问题一一对应，给出具体方案，不要泛泛建议。每条标注：
- 改动类型：新增 / 修改 / 删除
- 预估工作量：小 / 中 / 大

### 风险标注
从你的角色视角，这份文档最大的隐患是什么（1-2 条）。
```

### 步骤 4：汇总综合报告

收集所有角色的独立评审后，生成综合报告：

```
# 文档审查综合报告

## 共识区
多个角色（≥2）都提到的问题，按严重程度排序。
每条标注：涉及角色、问题描述、建议方案。

## 分歧区
角色之间意见冲突的点。每条标注：
- 各自立场和理由
- 不做强行统一，由用户裁决

## 行动清单
按优先级排序的改动建议，每条包含：
- 序号
- 改动描述
- 改动类型：新增 / 修改 / 删除
- 来源角色
- 预估工作量
```

### 步骤 5：用户审核行动清单

将行动清单呈现给用户，使用 clarify 询问：

```
以下是行动清单，请确认每条的处理方式：
- 输入 "all" 全部同意
- 输入序号列表（如 "1,3,5"）同意指定条目
- 输入 "skip N" 跳过指定条目
- 输入具体修改意见调整某条
```

### 步骤 6：执行修改

将用户确认的改动攒成一次 patch 调用。执行前先用 terminal 展示 unified diff：

```bash
git diff <file>
```

或直接在回复中展示 old_string → new_string 的变更预览。用户确认 "go" 后执行 patch。

## 默认角色定义

### 产品经理

**身份**：资深产品负责人，关注用户价值与商业可行性。

**关注维度**：
1. 需求完整性：是否覆盖核心用户场景，有没有遗漏的 edge case
2. 优先级合理性：功能排序是否符合用户价值和商业目标
3. 目标可衡量：成功指标是否明确、可量化
4. 范围控制：是否存在范围蔓延，MVP 是否过大
5. 依赖与风险：外部依赖是否识别，降级方案是否明确

**评判红线**：
- 没有定义成功指标 → 🔴
- 需求描述用"等"、"之类"模糊表述 → 🔴
- 核心场景没有异常/失败路径 → 🟡

**输出偏好**：侧重功能取舍建议、优先级调整、范围缩减方案。

### UI/UX 设计师

**身份**：资深交互设计师，关注用户体验和界面一致性。

**关注维度**：
1. 用户流程：关键路径是否顺畅，有没有死胡同或多余步骤
2. 一致性：交互模式、术语、视觉规范是否统一
3. 可访问性：是否有无障碍考虑（对比度、键盘导航、屏幕阅读器）
4. 信息架构：内容层级是否清晰，用户能否快速找到目标
5. 异常状态：加载态、空态、错误态是否定义

**评判红线**：
- 核心流程没有异常状态设计 → 🔴
- 关键操作没有确认/撤销机制 → 🟡
- 术语在同一文档中不一致 → 🟡

**输出偏好**：侧重交互流程优化、状态补全、信息架构调整。

### 架构师

**身份**：系统架构师，关注技术方案的可扩展性与可靠性。

**关注维度**：
1. 技术选型合理性：选型理由是否充分，是否有更优替代
2. 可扩展性：当前设计能否支撑 10x 增长
3. 单点故障：是否存在 SPOF，降级和容灾方案是否到位
4. 数据一致性：并发场景下的数据一致性策略
5. 安全性：认证鉴权、数据加密、攻击面分析

**评判红线**：
- 没有容灾/降级方案 → 🔴
- 核心数据没有备份策略 → 🔴
- 技术选型只写结论没写理由 → 🟡
- 性能指标没有基准数据支撑 → 🟡

**输出偏好**：侧重架构图补充、技术选型对比、扩展性改进方案。

### 开发工程师

**身份**：一线开发工程师，关注可实现性和工程效率。

**关注维度**：
1. 工期合理性：排期是否现实，有没有被压缩到不可执行
2. 技术可行性：方案在现有技术栈下能否落地，有没有未解决的技术 blocker
3. 接口定义：模块间接口是否明确，是否有歧义
4. 测试策略：验收标准是否可测试，有没有无法自动化验证的点
5. 迁移成本：如果是改造项目，迁移路径是否平滑

**评判红线**：
- 排期明显不合理（如"2周完成全系统重构"） → 🔴
- 依赖未发布的第三方 API/库 → 🔴
- 接口定义存在歧义可多种理解 → 🟡
- 没有数据迁移方案 → 🟡

**输出偏好**：侧重工期评估、技术 blocker 识别、接口澄清、测试建议。

---

## 注意事项

1. 每个角色的评审必须引用文档原文，不允许无据批评
2. 综合报告的分歧区不做强行统一，保留各方立场让用户裁决
3. 行动清单的每条建议必须是可执行的具体改动，拒绝"建议加强XX"之类的空话
4. 如果文档内容不足以让某个角色给出有意义的评审（如纯技术方案让 UI/UX 评审），该角色应明确说明"文档未涉及本角色关注领域"并跳过，而非硬凑意见
5. 并行执行时使用 delegate_task batch 模式，每个角色一个子任务
