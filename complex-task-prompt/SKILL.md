# Complex Task Prompt Generator

## Overview

对复杂任务进行结构化分析，用7模块提示词框架生成优化后的完整 prompt，经用户确认后再执行。核心理念：先打磨工具（prompt），再用工具干活。

## When to Use

自动触发条件（满足任一即触发）：
- 用户给出的任务包含 ≥3 个独立子目标
- 预计需要 ≥3 次工具调用
- 涉及跨领域知识（如：数据分析+前端展示+部署）
- 用户明确标注为"复杂任务"

**不触发：**
- 单一明确指令（"帮我查一下 XX"）
- 紧急修复（"这个报错怎么修"）
- 简单问答

## The 7-Module Framework

分析任务时，逐一填写以下7个模块，然后整合为一段完整 prompt。

### 1. Role（角色）— 1-2句话
定义模型的功能上下文。不是泛泛的"你是一个助手"，而是具体到这个任务需要什么能力。

```
你是一名 [具体领域] 专家，擅长 [核心能力]。
```

### 2. Personality（人设）
语气、风格、协作方式。控制输出的"味道"。

```
语气：[直接/严谨/探索性/...]
风格：[技术文档式/对话式/分析报告式/...]
协作：[独立执行/主动确认关键节点/...]
```

### 3. Goal（目标）
用户可见的最终产出。必须具体、可交付。

```
最终产出：[具体交付物描述]
交付形式：[文件/代码/报告/...]
```

### 4. Success（成功标准）
什么条件算完成。越具体越好，避免"做好了"这种模糊标准。

```
- [具体验收条件1]
- [具体验收条件2]
- [具体验收条件3]
```

### 5. Constraints（约束）
安全、商业、证据、副作用的边界。什么不能做、什么必须做。

```
禁止：[不能做的事情]
必须：[必须遵守的规则]
证据：[需要验证的要求]
```

### 6. Output（输出）
格式、长度、语气的具体约束。

```
格式：[Markdown/代码块/表格/...]
长度：[简短摘要/完整报告/按需]
语气：[客观中立/技术严谨/...]
```

### 7. Stop Rules（停止规则）
什么时候重试、回退、拒绝、追问或停止。

```
重试条件：[什么情况下重试]
回退策略：[失败时的降级方案]
拒绝条件：[什么情况下拒绝执行]
追问条件：[什么信息缺失需要追问]
停止条件：[什么情况下宣布任务完成]
```

## Workflow

### Step 1: 分析任务
读取用户输入，判断是否满足触发条件。

### Step 2: 填写7模块
对每个模块进行具体分析，结合任务上下文填写。关键原则：
- **每段保持简短**，只写能改变模型行为的重要信息
- **不写废话**，"你是AI助手"这种无信息量的内容直接删掉
- **约束要硬**，"尽量"、"最好"这种软约束不起作用

### Step 3: 整合为完整 prompt
将7个模块整合为一段连贯的、可直接使用的完整 prompt。整合规则：
- 按 Role → Personality → Goal → Success → Constraints → Output → Stop Rules 顺序排列
- 模块间用自然过渡连接，不要生硬拼接
- 总长度控制在 300-800 字（中文），过长反而稀释关键信息

### Step 4: 呈现给用户确认
输出格式：

```
📋 任务分析完成，以下是优化后的执行提示词：

[完整 prompt 内容]

---
请确认以上提示词是否准确表达了你的意图。
确认后我将直接按此执行。如需调整请指出具体模块。
```

### Step 5: 确认后执行
用户确认后，严格按 prompt 中定义的规则执行任务。执行过程中如遇到 prompt 未覆盖的情况，按 Stop Rules 处理。

## Prompt Template（快速填充）

```
## 角色
你是[领域]专家，擅长[能力]。

## 人设
语气[风格]，风格[形式]，遇到关键节点[协作方式]。

## 目标
完成[具体产出]，交付[形式]。

## 成功标准
- [验收条件1]
- [验收条件2]
- [验收条件3]

## 约束
禁止：[边界]
必须：[硬规则]
证据：[验证要求]

## 输出
格式[具体格式]，长度[要求]，语气[要求]。

## 停止规则
重试：[条件]
回退：[方案]
拒绝：[条件]
追问：[缺失信息]
停止：[完成标志]
```

## Common Pitfalls

1. **角色定义太泛** — "你是一个有用的助手"改变不了任何行为，必须具体到能力维度
2. **成功标准模糊** — "做好"不算标准，"输出包含 X、Y、Z 三个部分"才算
3. **约束太软** — "尽量避免"不会被遵守，"禁止"才会
4. **停止规则缺失** — 没有停止规则，模型会在死循环里浪费 token
5. **prompt 过长** — 超过 1000 字的 prompt 关键信息被稀释，核心指令反而不突出
6. **跳过确认直接执行** — 必须等用户确认，这是硬性规则

## Verification Checklist

- [ ] 7个模块全部填写，无遗漏
- [ ] 每个模块内容具体，无泛泛之谈
- [ ] 成功标准可量化或可验证
- [ ] 约束使用硬性语言（禁止/必须/不得）
- [ ] Stop Rules 覆盖重试、回退、拒绝、追问、停止5种场景
- [ ] 整合后的 prompt 长度在 300-800 字范围内
- [ ] 已呈现给用户等待确认
- [ ] 确认后才开始执行
